# Data Model and API Contracts

Date: 2026-03-18

## Product Entities

### Project

A `Project` is the durable user-facing container for work.

Fields:

- `project_id`
- `owner_id`
- `title`
- `goal`
- `status`
- `summary`
- `workspace_uri`
- `artifact_root`
- `default_worker_profile`
- `created_at`
- `updated_at`
- `last_activity_at`

### Worker

A `Worker` is an addressable execution identity inside a project.

Fields:

- `worker_id`
- `project_id`
- `owner_id`
- `name`
- `role`
- `profile`
- `backend`
- `state`
- `capabilities`
- `sandbox_profile`
- `session_ref`
- `takeover_url`
- `last_heartbeat_at`
- `created_at`
- `updated_at`

### Run

A `Run` is a concrete work attempt performed by a worker.

Fields:

- `run_id`
- `worker_id`
- `project_id`
- `instruction`
- `state`
- `started_at`
- `ended_at`
- `result_summary`
- `failure_reason`
- `metrics_json`

### Event

An `Event` is the canonical timeline record.

Fields:

- `event_id`
- `project_id`
- `worker_id`
- `run_id`
- `event_type`
- `message`
- `payload_json`
- `created_at`

### Artifact

An `Artifact` is a durable output or reference generated by a worker.

Fields:

- `artifact_id`
- `project_id`
- `worker_id`
- `run_id`
- `artifact_type`
- `path_or_url`
- `display_name`
- `metadata_json`
- `created_at`

## Backend Adapter Contract

Each adapter must implement the following logical operations:

- `create_worker(project, worker_spec)`
- `start_worker(worker)`
- `assign_run(worker, instruction)`
- `interrupt_run(worker, run)`
- `pause_worker(worker)`
- `resume_worker(worker)`
- `terminate_worker(worker)`
- `fetch_status(worker)`
- `fetch_events(worker, since_cursor)`
- `fetch_artifacts(worker, run)`
- `get_takeover(worker)`
- `reconcile_worker(worker)`

## Example Control Flows

### Create project and spawn worker

1. `POST /v1/projects`
2. `POST /v1/projects/{project_id}/workers`
3. `POST /v1/workers/{worker_id}/assign`
4. subscribe to `/events`

### Interrupt and redirect

1. `POST /v1/workers/{worker_id}/interrupt`
2. wait for state `paused` or `ready`
3. `POST /v1/workers/{worker_id}/message`
4. `POST /v1/workers/{worker_id}/resume` if needed

### Open worker control surface

1. `GET /v1/workers/{worker_id}/takeover`
2. open returned URL
3. operator acts
4. return control to system

## Notes On Honest Semantics

- `interrupt` means stop current work as quickly and safely as possible.
- `pause` means preserve resumable state when backend supports it.
- `terminate` means destroy the worker execution identity and release resources.
- `resume` means continue from persisted session or warm state, not create a fresh worker unless explicitly documented as fallback.
